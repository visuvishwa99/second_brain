# BigQuery Architecture
## Core Concept
* RDBMS vs BigQuery
  * RDBMS: Store + compute bundled together
  * BigQuery: Decoupling store and compute for individual pricing

## Components Mnemonic
* MN => Jupiter (gladiator) performs in Colossus with weapon called Dremel
  * Storage => Colossus
  * Compute => Dremel
  * Connecting Component => Jupiter
  * Orchestrator => Borg

## Storage Layer
* Colossus
  * Google's distributed file system
  * Columnar storage format for optimized queries
  * Automatic management and optimization

## Compute Layer
* Dremel: Query execution engine
  * Root server: Coordinates query execution
  * Mixers: Aggregate data
  * Leaf nodes: Read data from storage

## Connecting Components
* Jupiter: High-speed network protocol
  * Connects Dremel (compute) and Colossus (storage)
* Borg: Orchestrator
  * Manages internal communication between components

## Query Execution Flow
* Query Submission
  * User submits SQL query
* Query Parsing & Optimization
  * Root server creates execution plan
  * Applies cost-based optimizations
    * Predicate pushdown
    * Join reordering
    * Pruning unused columns
* Resource Allocation
  * Borg allocates compute slots
* Distributed Execution
  * Query broken into stages
  * Leaf nodes perform parallel scans
* Columnar Storage Access
  * Reads only required columns
  * Uses Capacitor format
* Aggregation & Finalization
  * Mixers aggregate intermediate results
  * Root server assembles final output
* Result Delivery
  * Final result returned to user

## Table Management
* DB => Dataset in BigQuery
* Empty tables
  * Created with defined schema, no data
  * Structure for future data insertion

## Partitioning
* Types of Partitions
  * Integer Range
  * Time-unit Column
  * Ingestion time
* Limitations
  * Cannot create string as partition column
