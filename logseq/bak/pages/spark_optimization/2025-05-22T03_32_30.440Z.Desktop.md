- What is Z-order clustering?
	- #card
	  id:: 6816857a-cbc8-45d6-87b9-6eeacdfc0687
		- Z-order clustering (also called Z-order indexing or Z-order curve) is a data organization technique that arranges multi-dimensional data  to a one-dimensional space while keeping related data points close to each other
		- The technique gets its name from the Z-shaped pattern it creates when visualized in two dimensions
		- Use partition by and Z-order both 
		  ```sql
		  # BETTER PRACTICE: Partition by date (lower cardinality)
		  spark.sql(f"""
		  CREATE TABLE sales_transactions_good ({sales_schema})
		  USING DELTA
		  PARTITIONED BY (YEAR(transaction_date), MONTH(transaction_date)) 
		  LOCATION '/data/sales_transactions_good'
		  """)
		  
		  # Then apply Z-Order within partitions
		  deltaTable = DeltaTable.forPath(spark, "/data/sales_transactions_good")
		  deltaTable.optimize().executeZOrderBy("store_id", "product_id")
		  
		  For datasets under 1TB:
		  
		  Start with no partitioning
		  Apply Z-Order based on most common query filters
		  Example: Product catalog Z-Ordered by category, brand
		  
		  
		  For larger time-series data:
		  
		  Partition by time at an appropriate granularity (year/month/day)
		  Apply Z-Order on high-cardinality columns within partitions
		  Example: Sales data partitioned by month, Z-Ordered by customer_id and product_id
		  ```

- What is data skew and how can it impact Spark performance?
	- #card
	  id:: 68168579-f03b-4580-b209-7585f0f72170
		- **Key techniques:**
		- **Salting**: Add random suffixes to keys to distribute processing (e.g., `concat($"key", lit("_"), (rand()*10).cast("int").cast("string"))`)
		- **Custom partitioning**: Implement specialized `Partitioner` classes to handle known skewed keys
		- **Broadcasting**: Use `broadcast(smallDF)` for joins when one dataset is significantly smaller
		- **Repartitioning**: Apply `repartition(n, $"skewed_column")` with higher partition counts
		- **Skewed key isolation**: Process highly skewed keys separately from normal data
		- **Adaptive Query Execution (AQE)**: Enable with `spark.sql.adaptive.skewJoin.enabled=true` in Spark 3.0+

- How is Liquid clustering,partitions and Z-Order Indexing implemented in Databricks and what is its syntax? Can you change the exisitng paritions ,Liquid clustering and Zorder?
	- #card
	  id:: 68168579-1006-4ecc-a107-1a5dc06d33cb
		- Z-Order Indexing is a multi-dimensional clustering technique that improves query performance with WHERE filters:
		  ```sql
		  
		  ---
		  ### 1. **Partitioning**
		  - Use `.partitionBy("sales_date")` when your data has a **low-cardinality column** that is frequently filtered (such as dates in sales data).
		  - Partitioning organizes data into directory structures, enabling efficient query pruning and faster reads. It also reduces the number of small files by grouping related records together.
		  - **Note:** Delta Lake does not support direct modification of partition columns for existing tables; to change partitions, you must rewrite the table with the new partitioning scheme.
		  
		  ---
		  
		  ### 2. **Z-Ordering**
		  - Apply `OPTIMIZE ... ZORDER BY (customer_id, product_id)` to cluster data within partitions based on columns that are often filtered but not partitioned (e.g., customer or product IDs).
		  - Z-Ordering improves data skipping and query performance by preserving the `min` and `max` values for the Z-ordered columns in each file, similar to how Snowflake organizes micro-partitions.
		  - **Examples:**
		   
		   -- Single column
		   OPTIMIZE customers ZORDER BY (customer_id);
		  
		   -- Multiple columns
		   OPTIMIZE sales_fact ZORDER BY (order_date, region, product_category);
		  
		   -- With table path
		   OPTIMIZE delta.`/path/to/table` ZORDER BY (user_id, event_date);
		   
		  - **Auto-optimize example:**
		   
		   ALTER TABLE sales_fact SET TBLPROPERTIES (
		  delta.autoOptimize.optimizeWrite = true,
		  delta.autoOptimize.autoCompact = true
		   );
		   
		  - **Note:** To update Z-Order columns, re-optimize the table with the new columns:  
		   `delta_table.optimize().executeZOrderBy("new_zorder_column1", "new_zorder_column2")`
		  
		  ---
		  
		  ### 3. **Liquid Clustering**
		  - Use `ALTER TABLE ... CLUSTER BY (device_id, event_type)` for **high-cardinality columns** or when your query/filter patterns may change over time.
		  - Liquid clustering allows you to flexibly and incrementally reorganize the data layout without rewriting the entire table, making it ideal for evolving analytics needs.
		  - To change the clustering columns, simply run another `ALTER TABLE ... CLUSTER BY (...)` statement.
		  
		  ---
		  
		  **Summary Table**
		  
		  | Technique           | Example Code / SQL                                     | When to Use                                              |
		  |---------------------|-------------------------------------------------------|----------------------------------------------------------|
		  | Partitioning        | `.partitionBy("sales_date")`                          | Low-cardinality, frequently filtered columns             |
		  | Z-Ordering          | `OPTIMIZE ... ZORDER BY (customer_id, product_id)`    | Frequent filters on non-partitioned columns              |
		  | Liquid Clustering   | `ALTER TABLE ... CLUSTER BY (device_id, event_type)`  | High-cardinality, evolving query/filter patterns         |
		  
		  ---
		  
		  **Key Notes:**
		  - Partition columns can’t be changed directly; rewriting the table is required.
		  - Z-Order columns can be changed by re-optimizing the table.
		  - Liquid clustering keys can be changed at any time with `ALTER TABLE ... CLUSTER BY`.
		  ```

- How do stages and tasks relate to jobs in Spark's execution model?
	- #card
	  id:: 68168579-3d82-4b9a-aee9-5efae587f63f
	    ```sql
	  1 Job → N Stages → M Tasks per Stage
	  	
	  Execution Flow
	  
	  Action triggers job creation
	  DAG Scheduler divides job into stages
	  Each stage is divided into tasks
	  Tasks are scheduled on executors
	  Results are collected back to driver
	  
	  import org.apache.spark.sql.SparkSession
	  import org.apache.spark.sql.functions._
	  
	  // Create SparkSession
	  val spark = SparkSession.builder()
	   .appName("Order Analysis")
	   .master("local[4]")  // 4 cores
	   .getOrCreate()
	  
	  // Sample data
	  val orders = Seq(
	    (1, "Electronics", 200, "CA"),
	    (2, "Books", 50, "NY"),
	    (3, "Electronics", 300, "CA"),
	    (4, "Clothing", 100, "TX"),
	    (5, "Books", 75, "CA"),
	    (6, "Electronics", 150, "NY"),
	    (7, "Clothing", 200, "CA"),
	    (8, "Books", 40, "TX")
	  ).toDF("order_id", "category", "amount", "state")
	  
	  // Partition into 2 partitions
	  val partitionedOrders = orders.repartition(2)
	  
	  // Complex query that creates multiple stages
	  val result = partitionedOrders
	    .filter($"amount" > 50)                    // Transformation 1
	    .withColumn("tax", $"amount" * 0.08)       // Transformation 2
	    .groupBy($"state", $"category")            // Transformation 3 (WIDE DEPENDENCY)
	    .agg(                                      // Transformation 4
	  	sum($"amount").alias("total_amount"),
	  	count("*").alias("order_count")
	    )
	    .orderBy($"state", $"total_amount".desc)   // Transformation 5 (WIDE DEPENDENCY)
	  
	  // ACTION - triggers the entire job
	  result.collect()
	  
	  Stage 0 (Narrow Dependencies Only)
	  ├── read data
	  ├── filter($"amount" > 50)           [Narrow - no shuffle]
	  ├── withColumn("tax", ...)           [Narrow - no shuffle]
	  └── (prepares data for groupBy)
	  	 │
	  	 │ SHUFFLE BOUNDARY
	  	 ▼
	  Stage 1 (After First Wide Dependency)
	  ├── groupBy($"state", $"category")   [WIDE - requires shuffle]
	  ├── agg(sum($"amount"))              [Narrow - part of groupBy]
	  └── (prepares data for ordering)
	  	 │
	  	 │ SHUFFLE BOUNDARY
	  	 ▼
	  Stage 2 (After Second Wide Dependency)
	  ├── orderBy($"state", ...)           [WIDE - requires shuffle]
	  └── collect()                        [Action - triggers job]
	    ```
