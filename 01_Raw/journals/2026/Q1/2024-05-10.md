 28. Statefull Vs Stateless Aggregation
	- These are two diff type of transformation used in streaming
	- Statefull Transformation -> spark is maintaining/storing the results . pass the results across the micro batches.
	 collapsed:: true
		- Ex: grouping, aggregation, windowing, joins
		- Excessive state may cause OOM
		- Managed Stateful Aggregations (managed by spark)
		- Custom Stateless Aggregations (by user)
		- **Examples:**
		- **Windowed Aggregations:** Calculating metrics (sum, average, count) over sliding or tumbling windows of time.
		- **Sessionization:** Grouping events into user sessions based on inactivity gaps.
		- **Stream-Stream Joins:** Joining two streams based on matching keys within a time window.
		- **Machine Learning:** Continuously updating models with new data (e.g., anomaly detection, recommendation systems).
		- **Business Use Cases:**
		- **Real-time Anomaly Detection:** Detecting unusual patterns or deviations from the norm in network traffic, sensor data, or financial transactions.
		- **Customer Behavior Analysis:** Identifying trends in customer behavior over time, like purchase patterns or product interactions.
		- **Personalized Recommendations:** Providing real-time recommendations based on user activity and preferences.
		- **Pros:**
		- **Rich Context:** Enables analysis that considers historical data and temporal relationships.
		- **Advanced Analytics:** Can power complex computations and machine learning models.
		- **Cons:**
		- **Complexity:** More challenging to implement and manage state.
		- **Performance Overhead:** Can be slower due to state management and potential checkpointing operations.
	- Stateless Transformation -> No need to store the state of results . so there are stateless
	 collapsed:: true
		- Implementation: Instead of taking results from "state store" we take those results from already exist target query and rewrite the merge statement .
		- We are removing rockdb config.
		- select(), filter(), map(), flatMap(), explode(), etc.
		- It does not support Complete output mode because they do not maintain state.
		- Few more Examples ->
			- Data Cleansing:** Filtering out invalid records, removing duplicates, handling null values
			- **Data Enrichment:** Joining streaming data with static lookup tables to add additional context.
			- **Simple Aggregations:** Counting the number of events within each micro-batch.
		- usecase : **Real-time Fraud Detection:** Quickly identifying suspicious transactions by applying rules to each transaction independently.
			- **Log Analysis:** Filtering and categorizing log messages in real-time for quick anomaly detection or alerting
			- **Pros:**
			- **Simplicity:** Easier to implement and reason about.
			- **Scalability:** Stateless transformations are highly parallelizable and can easily scale with the input data rate.
			- **Low Latency:** Typically have lower latency than stateful transformations.
			- **Cons:**
			- **Limited Context:** Cannot perform computations that require information from past events or maintain aggregates over time.

			- 31. Watermarking and State Store Cleanup / Late coming records handling
				- ![image.png](../assets/image_1715553885247_0.png)
				- Assume we created windows every 15 mins. Here after 10.45 (assume 10.46) we got two records
				 1) 10.48 2) 10.25 (Late record)
					- spark structure streaming can handle late record because the state store has information of previous window i.e 10.25 record belongs to 10:15 - 10.30 window.
					- so spark will fetch that previous record in state store and try to aggregate the late coming record .
					- 10.48 will be new window .
					- Statestore size increases because of as result. so expire timeframe is required (1d,1w) . This is called "Water marking".
					- What is a watermark?
					 How long do you want to wait for a late record.
					- How to decide your watermark
						- What is the maximum possible delay? Discuss with business
						- When late records are not relevant? After watermark timeframe
